1. PROJECT OVERVIEW — What is this project?
================================================================================

This is an Airline Booking System built with a true microservices architecture.
The system is now split into four independent services, each with its own
process, dependencies, and database:

  1. Flight Service   (Port 3000, DB: Flights)
    - Manages airplanes, flights, airports, cities
    - Provides advanced flight filtering and seat inventory updates

  2. Booking Service  (Port 3002, DB: Booking_DB)
    - Creates and cancels bookings
    - Calls Flight Service to verify availability and update seats

  3. Auth Service     (Port 3003, DB: Auth_DB)
    - Handles signup, login, and JWT token verification

  4. API Gateway      (Port 3001, no DB)
    - Single entry point for clients
    - Verifies JWT tokens and routes traffic to services

Note: The original monolith lives in the `node_Learning` folder for reference.
The active microservices live in `Airline-Booking-System/`.

2. ARCHITECTURE — How is the code organized?
================================================================================

The code follows a LAYERED ARCHITECTURE pattern. Each layer has ONE job:

  ┌─────────────────────────────────────────────────────────────────┐
  │                        CLIENT (Postman, Browser, Frontend)      │
  │                        Sends HTTP requests to the server        │
  └─────────────────────┬───────────────────────────────────────────┘
                        │
                        ▼
  ┌─────────────────────────────────────────────────────────────────┐
  │  ROUTES LAYER  (src/routes/)                                    │
  │  Job: Map URLs to the right controller function                 │
  │  Example: POST /api/v1/airplanes → createAirplane()             │
  └─────────────────────┬───────────────────────────────────────────┘
                        │
                        ▼
  ┌─────────────────────────────────────────────────────────────────┐
  │  MIDDLEWARE LAYER  (src/middlewares/)                            │
  │  Job: Validate the request BEFORE it reaches the controller     │
  │  Example: Check that "capacity" is a positive integer           │
  │  If invalid → returns 400 Bad Request, controller never runs    │
  │  If valid   → calls next(), request continues to controller     │
  └─────────────────────┬───────────────────────────────────────────┘
                        │
                        ▼
  ┌─────────────────────────────────────────────────────────────────┐
  │  CONTROLLER LAYER  (src/controllers/)                           │
  │  Job: Handle HTTP — extract data from request, call service,    │
  │       format the response with proper status code               │
  │  Does NOT contain business logic or database queries            │
  └─────────────────────┬───────────────────────────────────────────┘
                        │
                        ▼
  ┌─────────────────────────────────────────────────────────────────┐
  │  SERVICE LAYER  (src/services/)                                 │
  │  Job: Business logic. Currently thin (pass-through), but will   │
  │       grow as rules are added (e.g., "can't create duplicate    │
  │       airplane", "capacity must be within manufacturer specs")  │
  └─────────────────────┬───────────────────────────────────────────┘
                        │
                        ▼
  ┌─────────────────────────────────────────────────────────────────┐
  │  REPOSITORY LAYER  (src/repositories/)                          │
  │  Job: Talk to the database. Translate app operations into       │
  │       Sequelize ORM calls (which become SQL queries)            │
  └─────────────────────┬───────────────────────────────────────────┘
                        │
                        ▼
  ┌─────────────────────────────────────────────────────────────────┐
  │  MODEL LAYER  (src/models/)                                     │
  │  Job: Define the database table structure. What columns exist,  │
  │       what types they are, what constraints they have            │
  └─────────────────────┬───────────────────────────────────────────┘
                        │
                        ▼
  ┌─────────────────────────────────────────────────────────────────┐
  │  DATABASE  (MySQL)                                              │
  │  The actual tables where data is stored on disk                 │
  └─────────────────────────────────────────────────────────────────┘

WHY LAYERS?
  - Each layer can be changed independently without breaking others
  - You can swap MySQL for PostgreSQL by only changing the repository/model
  - You can add business rules without touching the database code
  - Testing is easier — you can test each layer in isolation

3. THE REQUEST LIFECYCLE — What happens when a request comes in?
================================================================================

Let's trace a COMPLETE request from client to database and back.

EXAMPLE: POST /api/v1/airplanes  with body: { "modelNumber": "A320", "capacity": 180 }

STEP 1 — Express receives the request
  File: src/index.js
  Line: app.use('/api', apirouter)
  
  Express sees the URL starts with "/api", so it passes the request
  to the "apirouter" (which is src/routes/index.js).
  
  The remaining URL after stripping "/api" is: /v1/airplanes

STEP 2 — Routes layer matches the URL
  File: src/routes/index.js
  Line: router.use('/v1', v1routes)
  
  Express sees the URL starts with "/v1", strips it, passes to v1routes.
  Remaining URL: /airplanes
  
  File: src/routes/v1/index.js
  Line: router.use('/airplanes', airplaneRoutes)
  
  Express sees "/airplanes", strips it, passes to airplaneRoutes.
  Remaining URL: /  (just the root)
  
  File: src/routes/v1/airplane-routes.js
  Line: router.post('/', AirplaneMiddleware.validateCreateRequest, Airplanecontroller.createAirplane)
  
  Express matches: method is POST, path is "/". 
  It sees TWO functions to run in order:
    1. AirplaneMiddleware.validateCreateRequest  (middleware)
    2. Airplanecontroller.createAirplane         (controller)

STEP 3 — Middleware validates the request
  File: src/middlewares/airplane-middleware.js
  Function: validateCreateRequest(req, res, next)
  
  Checks req.body.modelNumber → "A320" exists → PASS
  Checks req.body.capacity → 180 is a positive integer → PASS
  Logs: "Middleware: validateCreateRequest passed"
  Calls next() → Express runs the next function (the controller)

STEP 4 — Controller processes the request
  File: src/controllers/airplane-controller.js
  Function: createAirplane(req, res)
  
  Logs: "Controller: createAirplane request received"
  Extracts data from req.body: { modelNumber: "A320", capacity: 180 }
  Calls: AirplaneService.createAirplane({ modelNumber: "A320", capacity: 180 })
  Waits for the result...

STEP 5 — Service runs business logic
  File: src/services/airplaneService.js
  Function: createAirplane(data)
  
  Logs: "Airplane service: createAirplane called"
  Currently no business logic — just passes data to repository.
  Calls: airplaneRepository.create(data)
  Waits for the result...

STEP 6 — Repository talks to database
  File: src/repositories/crud-repository.js
  Function: create(data)
  
  Calls: this.model.create(data)
  "this.model" is the Airplane Sequelize model (set in constructor).
  
  Sequelize translates this into SQL:
    INSERT INTO `Airplanes` (`id`, `modelNumber`, `capacity`, `createdAt`, `updatedAt`)
    VALUES (DEFAULT, 'A320', 180, '2026-04-02 10:30:00', '2026-04-02 10:30:00');
  
  MySQL executes the SQL and returns the created row.
  Sequelize wraps it in an Airplane model instance.
  
  Logs: "Successfully created a resource"
  Returns the airplane object back to the service.

STEP 7 — Response travels back up

  Service receives the airplane → returns it to controller
  Controller receives the airplane → logs success → sends HTTP response:
  
  HTTP/1.1 201 Created
  Content-Type: application/json
  
  {
    "success": true,
    "message": "Successfully created an airplane",
    "data": {
      "id": 1,
      "modelNumber": "A320",
      "capacity": 180,
      "createdAt": "2026-04-02T10:30:00.000Z",
      "updatedAt": "2026-04-02T10:30:00.000Z"
    },
    "error": {}
  }

COMPLETE LOG OUTPUT for this request:
  info: Middleware: validateCreateRequest passed
  info: Controller: createAirplane request received {"body":{"modelNumber":"A320","capacity":180}}
  info: Airplane service: createAirplane called {"data":{"modelNumber":"A320","capacity":180}}
  info: Successfully created a resource
  info: Controller: createAirplane success {"id":1}

────────────────────────────────────────────────────────────────────────────────
FILE: src/index.js
PURPOSE: The ENTRY POINT of the entire application. Sets up Express, mounts
         routes, and starts listening for HTTP requests.
────────────────────────────────────────────────────────────────────────────────

LINE: const express = require('express');
  → Imports the Express library. Express is a web framework that creates
    an HTTP server and provides routing, middleware, and request/response
    handling. Without Express, you'd need to use Node's raw http module
    and manually parse URLs, headers, and body data.

LINE: const app = express();
  → Creates an Express APPLICATION instance. This object IS the server.
    It stores all routes, middleware, and settings. You attach everything
    to this object, then call app.listen() to start it.

LINE: const { serverConfig, Logger } = require('./config');
  → Imports from the config barrel export:
    - serverConfig: Contains PORT from .env file
    - Logger: The Winston logger instance (singleton — same object everywhere)
    
    DESTRUCTURING SYNTAX: { serverConfig, Logger } = require(...)
    This is JavaScript destructuring. It's equivalent to:
      const config = require('./config');
      const serverConfig = config.serverConfig;
      const Logger = config.Logger;

LINE: const apirouter = require('./routes');
  → Imports the main route handler from src/routes/index.js.
    This contains all sub-routes (v1/airplanes, v1/info, etc.)

LINE: app.use(express.json());
  → MIDDLEWARE: Tells Express to parse incoming JSON request bodies.
    When a client sends:
      POST /api/v1/airplanes
      Content-Type: application/json
      {"modelNumber": "A320", "capacity": 180}
    
    Express.json() parses that JSON string and puts the result in req.body.
    WITHOUT this line, req.body would be UNDEFINED.
    
    HOW IT WORKS INTERNALLY:
    1. Express checks the Content-Type header
    2. If it's "application/json", it reads the raw body (a Buffer)
    3. It calls JSON.parse() on the raw string
    4. It sets req.body = the parsed JavaScript object
    5. It calls next() to continue to the next middleware

LINE: app.use(express.urlencoded({extended:true}));
  → MIDDLEWARE: Parses URL-encoded form data (like HTML <form> submissions).
    When a client sends:
      POST /api/v1/airplanes
      Content-Type: application/x-www-form-urlencoded
      modelNumber=A320&capacity=180
    
    This middleware parses it into req.body = { modelNumber: "A320", capacity: "180" }
    
    {extended: true} means it uses the "qs" library for parsing, which supports
    nested objects like: user[name]=John → { user: { name: "John" } }
    If extended were false, it uses the simpler "querystring" parser.

LINE: app.use('/api', apirouter);
  → ROUTE MOUNTING: All routes defined in apirouter will be prefixed with "/api".
    So if apirouter has a route for "/v1/airplanes", the full URL becomes
    "/api/v1/airplanes".
    
    app.use() with a path prefix strips that prefix before passing to the router.
    When a request for "/api/v1/airplanes" comes in:
    1. Express sees it starts with "/api" → matches this middleware
    2. Strips "/api" from the URL → remaining: "/v1/airplanes"
    3. Passes the request (with shortened URL) to apirouter

LINE: app.listen(serverConfig.PORT, () => { ... });
  → STARTS THE SERVER. This tells Node.js to:
    1. Create a TCP server on the specified port
    2. Start accepting incoming HTTP connections
    3. Run the callback function when the server is ready
    
    The callback logs: "Server started successfully on port 3000"
    This only runs ONCE, when the server first boots.
    
    If PORT is undefined (no .env file), Node will pick a random port.
    If PORT is already in use, this will throw an EADDRINUSE error.

LINE: Logger.info(`Server started successfully on port ${serverConfig.PORT}`);
  → Uses template literal (backticks) to embed the port number in the message.
    This log goes to:
    - Console (colorized, green "info" prefix) — in development
    - logs/combined.log (JSON format with timestamp)

────────────────────────────────────────────────────────────────────────────────
FILE: src/config/logger-config.js
PURPOSE: Creates and configures a Winston logger instance used throughout
         the entire application for structured, leveled logging.
────────────────────────────────────────────────────────────────────────────────

LINE: const winston = require('winston');
  → Imports the Winston logging library. Winston provides:
    - Severity levels (error, warn, info, http, debug, verbose, silly)
    - Multiple output destinations (console, file, HTTP, database)
    - Customizable formatting (JSON, simple text, colorized)
    - Timestamp injection

LINE: const logger = winston.createLogger({ ... });
  → Creates a LOGGER INSTANCE. This is the main object you use to log.
    createLogger() takes a configuration object with:
    - level: minimum severity to log
    - format: how logs are formatted
    - transports: where logs are sent

  INSIDE THE CONFIG:

  level: 'info'
    → Sets the minimum severity. Only messages at this level or higher
      (more severe) are logged.
      
      Winston severity levels (built-in):
        error: 0  ← most severe (ALWAYS logged)
        warn:  1
        info:  2  ← this is our threshold
        http:  3  ← NOT logged (below threshold)
        verbose: 4 ← NOT logged
        debug: 5  ← NOT logged
        silly: 6  ← NOT logged (least severe)
      
      So Logger.info("hello")  → LOGGED
         Logger.error("crash") → LOGGED (more severe than info)
         Logger.debug("x=5")   → IGNORED (less severe than info)

  format: winston.format.combine(...)
    → combine() chains multiple format transformers. Each one modifies
      the log "info" object, then passes it to the next.
      
      winston.format.timestamp({ format: 'YYYY-MM-DD HH:mm:ss' })
        → Adds a "timestamp" field to every log entry.
          Before: { level: 'info', message: 'hello' }
          After:  { level: 'info', message: 'hello', timestamp: '2026-04-02 10:30:00' }
      
      winston.format.json()
        → Converts the entire log object to a JSON string for output.
          Output: {"level":"info","message":"hello","timestamp":"2026-04-02 10:30:00"}

  transports: [ ... ]
    → Transports are OUTPUT DESTINATIONS. Each transport independently
      receives the formatted log and writes it somewhere.
      
      new winston.transports.File({ filename: 'logs/combined.log' })
        → Writes ALL logs (at the configured level and above) to this file.
          The file is created automatically. It grows over time — each log
          is appended to the end. In production, you'd rotate these files
          (delete old ones) using winston-daily-rotate-file transport.
      
      new winston.transports.File({ filename: 'logs/error.log', level: 'error' })
        → Writes ONLY error-level logs to this file. The "level: 'error'"
          on this transport OVERRIDES the logger-level setting. Even though
          the logger is set to 'info', this transport only accepts 'error'.
          
          This is a common pattern: keep ALL logs in combined.log for
          full tracing, but also keep errors in a separate file for
          quick access when debugging production issues.

  CONDITIONAL CONSOLE TRANSPORT:
  
  if (process.env.NODE_ENV !== 'production') { ... }
    → In development, also log to the terminal console.
      In production, console output is often unnecessary because logs
      go to file-based or cloud-based aggregators.
      
      winston.format.colorize()
        → Colors the output based on level:
          error → red
          warn  → yellow  
          info  → green
          This makes it easy to scan the terminal for errors.
      
      winston.format.simple()
        → Formats output as: "level: message"
          Example: "info: Server started on port 3000"
          This is human-readable (unlike JSON format used in files).

LINE: module.exports = logger;
  → Exports the logger instance. Because Node.js CACHES module exports,
    every file that requires this module gets the SAME logger object.
    This is a SINGLETON pattern — one logger for the whole app.

FILE: src/models/index.js
PURPOSE: Auto-discovers and loads ALL Sequelize model files in the models/ folder.

────────────────────────────────────────────────────────────────────────────────

LINE: const fs = require('fs');
  → Node.js file system module. Used to read the models directory.

LINE: const path = require('path');
  → Node.js path module. Used to join file paths cross-platform.

LINE: const Sequelize = require('sequelize');
  → The Sequelize ORM library.

LINE: const basename = path.basename(__filename);
  → Gets the filename of THIS file: "index.js"
    Used later to skip itself when scanning for model files.

LINE: const env = process.env.NODE_ENV || 'development';
  → Determines which environment we're in. Defaults to 'development'.

LINE: const config = require(__dirname + '/../config/config.json')[env];
  → Loads the database config for the current environment.
    In development: { username: "root", password: "ankit0709", database: "Flights", ... }

LINE: const db = {};
  → An empty object that will hold all model classes. Eventually:
    db = { Airplane: AirplaneModel, sequelize: connection, Sequelize: library }

LINE: let sequelize;
  → Declares the database connection variable.

LINE: if (config.use_env_variable) { ... } else { ... }
  → Two ways to connect to the database:
    1. Via an environment variable name (e.g., DATABASE_URL) — used in production
    2. Via individual config values (username, password, etc.) — used in development
    Our config.json doesn't have use_env_variable, so we use option 2:
    
    sequelize = new Sequelize(config.database, config.username, config.password, config);
    → Creates a connection pool to MySQL. Does NOT connect yet — 
      connection happens on first query.
      Parameters: Sequelize('Flights', 'root', 'ankit0709', { host: '127.0.0.1', dialect: 'mysql' })

LINE: fs.readdirSync(__dirname).filter(...).forEach(...)
  → This is the AUTO-DISCOVERY mechanism:
    
    1. fs.readdirSync(__dirname)
       Reads all files in src/models/: ["airplane.js", "index.js"]
    
    2. .filter(file => ...)
       Filters out:
       - Files starting with "." (hidden files like .gitkeep)
       - This file itself ("index.js") — so it doesn't try to load itself
       - Non-.js files
       - Test files (*.test.js)
       Remaining: ["airplane.js"]
    
    3. .forEach(file => ...)
       For each model file:
       - require(path.join(__dirname, file)) → loads src/models/airplane.js
       - That file exports a FUNCTION: (sequelize, DataTypes) => Airplane class
       - We call that function: modelFunction(sequelize, Sequelize.DataTypes)
       - This returns the Airplane class, registered with our connection
       - We store it: db['Airplane'] = AirplaneClass
    
    WHY THIS PATTERN?
    When you add a new model (e.g., Flight), you just create the file.
    You don't need to manually import it anywhere — this loop finds it automatically.

LINE: Object.keys(db).forEach(modelName => { if (db[modelName].associate) ... })
  → Runs the "associate" method on every model (if it exists).
    This is where you define RELATIONSHIPS between models:
    - Airplane.hasMany(Flight)
    - Flight.belongsTo(Airplane)
    Currently, Airplane.associate() is empty (no relationships yet).

LINE: db.sequelize = sequelize;
  → Stores the connection instance so other files can use it for
    raw queries, transactions, etc.

LINE: db.Sequelize = Sequelize;
  → Stores the Sequelize library itself (useful for accessing DataTypes,
    Op operators, etc.)

LINE: module.exports = db;
  → Exports: { Airplane: AirplaneModel, sequelize: connection, Sequelize: library }
    Other files import specific models: const { Airplane } = require('../models');


────────────────────────────────────────────────────────────────────────────────
FILE: src/models/airplane.js
PURPOSE: Defines the Airplane model — the JavaScript representation of the
         "Airplanes" table in MySQL.
────────────────────────────────────────────────────────────────────────────────

────────────────────────────────────────────────────────────────────────────────

WHAT IS A MIGRATION?
  A migration is a JavaScript file that describes ONE change to the database.
  It has TWO directions:
    - up():   Apply the change   (create the table)
    - down(): Reverse the change (drop the table)
  
  Migrations run IN ORDER (by timestamp in filename). Sequelize tracks which
  have run in a special table called "SequelizeMeta".
  
  WHY NOT JUST CREATE TABLES MANUALLY?
    - Team members need the same database structure
    - You can't "undo" a manual SQL change
    - In production, you need automated, repeatable deployments
    - Migrations are code — they go in Git, get reviewed, versioned

THE FILENAME: 20260329111835-create-airplane.js
  → 20260329111835 = year-month-day-hour-minute-second
    Sequelize runs migrations in chronological order by this timestamp.

LINE: async up(queryInterface, Sequelize) { ... }
  → The "forward" direction. Creates the table.
  
  queryInterface: A Sequelize object that provides methods to modify
    the database schema (createTable, addColumn, removeColumn, etc.)
  
  await queryInterface.createTable('Airplanes', { ... })
  → Generates and runs SQL:
    CREATE TABLE `Airplanes` (
      `id` INTEGER NOT NULL AUTO_INCREMENT PRIMARY KEY,
      `modelNumber` VARCHAR(255) NOT NULL,
      `capacity` INTEGER NOT NULL,
      `createdAt` DATETIME NOT NULL,
      `updatedAt` DATETIME NOT NULL
    );

  Column definitions:
    id:          Auto-incrementing primary key. Every row gets a unique ID.
    modelNumber: String, NOT NULL (required — can't insert without it).
    capacity:    Integer, NOT NULL (required).
    createdAt:   Automatically set by Sequelize when a row is created.
    updatedAt:   Automatically updated by Sequelize when a row is modified.

LINE: async down(queryInterface, Sequelize) { ... }
  → The "reverse" direction.
  
  await queryInterface.dropTable('Airplanes')
  → Generates: DROP TABLE `Airplanes`;
    Completely removes the table and all its data. DESTRUCTIVE.
    Used when you run: npm run migration:undo

FILE: src/middlewares/airplane-middleware.js
PURPOSE: Validates incoming request bodies BEFORE they reach the controller.
         Acts as a gatekeeper — rejects bad data with clear error messages.
────────────────────────────────────────────────────────────────────────────────

WHAT IS MIDDLEWARE?
  In Express, middleware is a function with signature: (req, res, next)
  It sits BETWEEN the route match and the controller.
  It can:
  1. Inspect/modify the request (req)
  2. Send a response (res) and STOP the chain
  3. Call next() to pass control to the next function

  Route definition: router.post('/', validateCreateRequest, createAirplane)
  → Express runs validateCreateRequest FIRST. If it calls next(), createAirplane runs.
    If it sends a response (res.status(400).json(...)), createAirplane NEVER runs.

LINE: const { StatusCodes } = require('http-status-codes');
  → Imports named HTTP status codes. StatusCodes.BAD_REQUEST = 400.
    Using names instead of numbers makes code more readable and less error-prone.

LINE: const { Logger } = require('../config');
  → Imports the Winston logger for logging validation results.

FUNCTION: validateCreateRequest(req, res, next)
  → Validates the body for POST /api/v1/airplanes (creating a new airplane).

  CHECK 1: if (!req.body.modelNumber)
    → The ! (NOT) operator. Truthy/falsy check.
      Returns true (fails validation) if modelNumber is:
      - undefined  (field not sent)
      - null
      - ""         (empty string)
      - 0          (zero — but model numbers shouldn't be 0)
      
      If it fails:
      - Logger.warn() → logs the bad request (warning level, not error)
      - Returns 400 Bad Request with explanation
      - Does NOT call next() → controller never runs
      - return statement ensures no further code in this function runs

  CHECK 2: if (!req.body.capacity || !Number.isInteger(req.body.capacity) || req.body.capacity <= 0)
    → Three conditions, ANY one failing rejects the request:
      
      !req.body.capacity
        → capacity is missing, null, 0, or undefined
      
      !Number.isInteger(req.body.capacity)
        → capacity exists but is not an integer.
          Number.isInteger(180)   → true  ✓
          Number.isInteger(180.5) → false ✗ (decimal)
          Number.isInteger("180") → false ✗ (string — JSON would parse as int though)
      
      req.body.capacity <= 0
        → capacity is zero or negative. An airplane with 0 seats makes no sense.
      
      Why || (OR)?
        If ANY check is true, the whole condition is true → validation fails.
        || short-circuits: if the first check is true, it doesn't evaluate the rest.

  LINE: next();
    → All checks passed. Tells Express to run the next function in the chain
      (which is the controller). Without calling next(), the request would
      hang forever — the client would never get a response.

FUNCTION: validateUpdateRequest(req, res, next)
  → Validates the body for PATCH /api/v1/airplanes/:id (updating an airplane).
  
  DIFFERENCE FROM CREATE:
  For updates, not ALL fields are required. You might want to update just
  the capacity without changing the modelNumber. So:
  
  CHECK 1: if (!req.body.modelNumber && !req.body.capacity)
    → BOTH are missing. There's nothing to update. That's an error.
      Using && (AND): BOTH must be falsy for this to fail.
      If at least one is present, this check passes.
  
  CHECK 2: if (req.body.capacity !== undefined && (invalid check))
    → Only validates capacity IF it was provided.
      req.body.capacity !== undefined: guards against validating absent fields.
      If you only sent { modelNumber: "B737" } with no capacity, 
      this check is skipped entirely.

LINE: const { Logger } = require("../config");
  → Imports the Winston logger.

FUNCTION: async function createAirplane(req, res) { ... }
  → Handles: POST /api/v1/airplanes
  
  req.body contains the parsed JSON body (parsed by express.json() middleware).
  Example: { modelNumber: "A320", capacity: 180 }
  
  FLOW:
  1. Logger.info('...request received', { body: req.body })
     → Logs what data came in. The second argument is METADATA — Winston

     → Status 500 INTERNAL SERVER ERROR means "something went wrong on our end."
       We return the error message but NOT the stack trace (security — don't
       expose internal details to clients).

  Returns 200 OK with the array.

FUNCTION: async function getAirplane(req, res) { ... }
  → Handles: GET /api/v1/airplanes/:id
  
  req.params.id contains the URL parameter.
  Example: GET /api/v1/airplanes/5 → req.params.id = "5" (a STRING)

  Sequelize's findByPk() can handle string "5" — it auto-converts.

    Logger.warn() → uses WARN level (not error — the app isn't broken,
    the client just asked for something that doesn't exist)

FUNCTION: async function updateAirplane(req, res) { ... }
  → Handles: PATCH /api/v1/airplanes/:id
  
  req.params.id = the ID from URL
  req.body = the fields to update
  
  WHY PATCH (not PUT)?
    PUT means "replace the entire resource with this new data."
    PATCH means "update only the specified fields."
    Since we allow partial updates (just capacity, or just modelNumber),
    PATCH is the correct HTTP method.
  
  Sequelize's update() returns [affectedCount]:
    [1] → one row was updated (success)
    [0] → no row with that ID exists
  
  response[0] === 0 check → returns 404 if no row was updated.

FUNCTION: async function destroyAirplane(req, res) { ... }

    1 → row was deleted
    0 → no row with that ID
  
  response === 0 check → returns 404 if nothing was deleted.
  
  NOTE: This is a HARD DELETE — the row is permanently removed from the
  database. In production systems, you might prefer SOFT DELETE (setting a
  "deletedAt" timestamp instead, so data can be recovered). Sequelize supports
  soft delete with the "paranoid: true" model option.

RESPONSE FORMAT — All endpoints return this consistent structure:
  {
    "success": true/false,     ← Quick boolean check for the client
    "message": "...",          ← Human-readable description of what happened
    "data": { ... } or [],     ← The actual data (airplane object, array, etc.)
    "error": {} or "..."       ← Error details (empty object on success)
  }
  
  WHY CONSISTENT FORMAT?
  The frontend can always expect the same structure. No guessing whether
  the response has "result", "data", "payload", etc. Every response has
  success + message + data + error.

8. ERROR HANDLING FLOW — How errors propagate through layers
================================================================================

Errors in this application follow a BUBBLE-UP pattern.
Each layer catches the error, logs it, then re-throws it to the layer above.
Only the CONTROLLER sends the HTTP response.

EXAMPLE: Database connection is down. Client sends POST /api/v1/airplanes.

STEP 1 — Repository layer:
  try {
    await this.model.create(data);           ← MySQL throws ECONNREFUSED
  } catch (error) {
    Logger.error('Error in create repo...')   ← Logged at REPO level
    throw error;                              ← Re-thrown to service
  }

STEP 2 — Service layer:
  try {
    await airplaneRepository.create(data);    ← Catches the re-thrown error
  } catch (error) {
    Logger.error('Airplane service: create failed')  ← Logged at SERVICE level
    throw error;                              ← Re-thrown to controller
  }

STEP 3 — Controller layer:
  try {
    await AirplaneService.createAirplane(data);  ← Catches the re-thrown error
  } catch (error) {
    Logger.error('Controller: create failed')     ← Logged at CONTROLLER level
    return res.status(500).json({                 ← SENDS THE HTTP RESPONSE
      success: false,
      message: 'Failed to create an airplane',
      error: error.message                        ← "connect ECONNREFUSED"
    });
  }

FULL LOG OUTPUT (all three layers logged):
  error: Error in create repository layer {"error":"connect ECONNREFUSED 127.0.0.1:3306"}
  error: Airplane service: createAirplane failed {"error":"connect ECONNREFUSED 127.0.0.1:3306"}
  error: Controller: createAirplane failed {"error":"connect ECONNREFUSED 127.0.0.1:3306"}

LOGS GO TO:
  Console → All three lines (red, because error level)
  logs/combined.log → All three lines (JSON format with timestamps)
  logs/error.log → All three lines (because level is 'error')

CLIENT RECEIVES:
  HTTP/1.1 500 Internal Server Error
  { "success": false, "message": "Failed to create...", "error": "connect ECONNREFUSED..." }

WHY LOG AT EVERY LAYER?
  Because each layer adds CONTEXT. The repo log says "database failed."
  The service log says "createAirplane failed." The controller log says
  "the POST /airplanes endpoint failed." Together they form a complete
  trace: which endpoint → which operation → which database call failed.
  In a large system with many models and operations, this trace is invaluable.

9. PHASE 3 — CITY & AIRPORT MODELS (Complete Documentation)
================================================================================

PHASE 3 OBJECTIVE: Add City and Airport models with a parent-child relationship
(City hasMany Airports, Airport belongsTo City), full CRUD for both, with
CASCADE deletes ensuring data integrity.


────────────────────────────────────────────────────────────────────────────────
CONCEPT: DATABASE RELATIONSHIPS (Foreign Keys)
────────────────────────────────────────────────────────────────────────────────

A FOREIGN KEY is a column in one table that references the PRIMARY KEY of
another table. It creates a LINK between two tables.

In our system:
  - The Cities table has an "id" column (primary key)
  - The Airports table has a "cityId" column (foreign key → Cities.id)

This means:
  1. Every airport MUST belong to a city (cityId cannot be null)
  2. You CANNOT insert an airport with a cityId that doesn't exist in Cities
     → MySQL rejects it with: ER_NO_REFERENCED_ROW
  3. If you delete a city, all its airports are automatically deleted (CASCADE)

WHY NOT JUST STORE THE CITY NAME ON THE AIRPORT?
  If you did: Airports = { name: "BOM", cityName: "Mumbai" }
  Problems:
  - If "Mumbai" is renamed to "Mumbai City", you must update EVERY airport row
  - No validation — you could typo "Mumba" and it would be accepted
  - You can't easily list "all airports in Mumbai" without string matching
  
  With a foreign key: Airports = { name: "BOM", cityId: 1 }
  Benefits:
  - Change city name in one place (Cities table), all airports reflect it
  - MySQL enforces that cityId must reference a real city
  - Query is simple: WHERE cityId = 1

CASCADE DELETE:
  When you delete a city, what should happen to its airports?
  Options:
  - RESTRICT: Prevent deletion if airports exist (error)
  - SET NULL: Set cityId to null on orphaned airports
  - CASCADE: Delete all airports that belonged to that city
  
  We chose CASCADE: Deleting "Mumbai" also deletes "BOM" and "JUH" airports.
  This makes sense because an airport cannot exist without a city.


────────────────────────────────────────────────────────────────────────────────
CONCEPT: SEQUELIZE ASSOCIATIONS (hasMany / belongsTo)
────────────────────────────────────────────────────────────────────────────────

Sequelize needs to know about relationships from BOTH sides:

  City.hasMany(Airport, { foreignKey: 'cityId', onDelete: 'CASCADE' })
    → "One City has many Airports. The link is via cityId column on Airport."
    → Enables: City.findAll({ include: Airport })
    → SQL: SELECT * FROM Cities LEFT JOIN Airports ON Cities.id = Airports.cityId

  Airport.belongsTo(City, { foreignKey: 'cityId', onDelete: 'CASCADE' })
    → "One Airport belongs to one City. My cityId column points to City."
    → Enables: Airport.findAll({ include: City })
    → SQL: SELECT * FROM Airports LEFT JOIN Cities ON Airports.cityId = Cities.id

WHERE ARE THESE DEFINED?
  In each model's static associate() method:
  
  // In src/models/city.js
  static associate(models) {
    this.hasMany(models.Airport, { foreignKey: 'cityId', onDelete: 'CASCADE' });
  }
  
  // In src/models/airport.js
  static associate(models) {
    this.belongsTo(models.City, { foreignKey: 'cityId', onDelete: 'CASCADE' });
  }

WHEN ARE THEY CALLED?
  In src/models/index.js, after all models are loaded:
  
  Object.keys(db).forEach(modelName => {
    if (db[modelName].associate) {
      db[modelName].associate(db);
    }
  });
  
  This loop calls City.associate(db) and Airport.associate(db),
  passing the db object (which contains ALL loaded models).
  Inside associate(), "models.Airport" resolves to the Airport class.

INCLUDE (Eager Loading):
  When fetching airports, we want the city data embedded:
  
  Airport.findAll({ include: [{ model: City, attributes: ['id', 'name'] }] })
  
  Generates SQL:
    SELECT Airports.*, Cities.id, Cities.name
    FROM Airports
    LEFT OUTER JOIN Cities ON Airports.cityId = Cities.id;
  
  Returns:
  [
    {
      id: 1, name: "CSMI Airport", code: "BOM", cityId: 1,
      City: { id: 1, name: "Mumbai" }    ← embedded city data!
    }
  ]
  
  "attributes: ['id', 'name']" limits which City columns are included.
  Without it, you'd also get createdAt, updatedAt for the City — unnecessary noise.

    on which class you're using.

─── MODEL: src/models/flight.js ───────────────────────────────────────────────

PURPOSE: Defines the Flight Sequelize model with 3 associations.

ASSOCIATIONS (belongsTo):
  1. belongsTo(Airplane, { foreignKey: 'airplaneId', onDelete: 'CASCADE' })
     → A flight uses one airplane.

  2. belongsTo(Airport, {
       foreignKey: 'departureAirportId',
       targetKey: 'code',            ← KEY: matches via Airport.code, not Airport.id
       as: 'departureAirport',       ← ALIAS: required because two FKs point to same table
       onDelete: 'CASCADE'
     })
     → A flight departs from one airport.

  3. belongsTo(Airport, {
       foreignKey: 'arrivalAirportId',
       targetKey: 'code',
       as: 'arrivalAirport',
       onDelete: 'CASCADE'
     })
     → A flight arrives at one airport.

WHY ALIASES ARE REQUIRED:
  Without 'as: departureAirport' and 'as: arrivalAirport', Sequelize wouldn't
  know which FK to use when eager-loading Airport data. The alias tells it:
    include: { model: Airport, as: 'departureAirport' }  → uses departureAirportId
    include: { model: Airport, as: 'arrivalAirport' }    → uses arrivalAirportId

WHY targetKey: 'code':
  Sequelize defaults to matching FKs against the target model's PRIMARY KEY (id).
  But our FK stores airport CODES (strings like "DEL"), not airport IDs.
  targetKey: 'code' tells Sequelize: "match this FK against Airport.code, not Airport.id".
  Without it, eager loading returns NULL (id=DEL doesn't exist).

─── REPOSITORY: src/repositories/flight-repository.js ─────────────────────────

PURPOSE: Extends CrudRepository with smart filtering and eager loading.

METHODS:
  constructor()     — calls super(Flight) to set this.model = Flight
  getAll(filter)    — OVERRIDDEN: dynamic WHERE clause + sorting + eager loading
  get(id)           — OVERRIDDEN: eager loading of Airplane + both Airports

INHERITED FROM CrudRepository (not overridden):
  create(data)      — Flight.create(data)
  update(data, id)  — Flight.update(data, { where: { id } })
  destroy(id)       — Flight.destroy({ where: { id } })

getAll(filter) — THE FILTERING ENGINE:
  This method builds a dynamic Sequelize WHERE clause based on optional filter params.

  STEP 1 — Route filtering:
    if (filter.departureAirportId) where.departureAirportId = filter.departureAirportId;
    if (filter.arrivalAirportId)  where.arrivalAirportId = filter.arrivalAirportId;
    SQL: WHERE departureAirportId = 'DEL' AND arrivalAirportId = 'BOM'

  STEP 2 — Price range:
    if (filter.minPrice) where.price[Op.gte] = filter.minPrice;  (≥)
    if (filter.maxPrice) where.price[Op.lte] = filter.maxPrice;  (≤)
    SQL: WHERE price >= 1000 AND price <= 5000

  STEP 3 — Departure date filtering:
    Creates a start-of-day (00:00:00) and end-of-day (23:59:59) range.
    where.departureTime = { [Op.between]: [startOfDay, endOfDay] }
    SQL: WHERE departureTime BETWEEN '2025-12-25 00:00:00' AND '2025-12-25 23:59:59'

  STEP 4 — Sorting:
    const sortField = filter.sort || 'departureTime';  (default sort)
    const sortOrder = filter.order || 'ASC';            (default direction)
    order: [[sortField, sortOrder]]

  STEP 5 — Eager loading (same for getAll and get):
    include: [
      { model: Airplane, attributes: ['id', 'modelNumber', 'capacity'] },
      { model: Airport, as: 'departureAirport', attributes: ['code', 'name', 'cityId'] },
      { model: Airport, as: 'arrivalAirport', attributes: ['code', 'name', 'cityId'] }
    ]

  EXAMPLE API CALL:
    GET /api/v1/flights?departureAirportId=DEL&arrivalAirportId=BOM&minPrice=1000&maxPrice=5000&sort=price&order=ASC

    This generates:
      SELECT * FROM Flights
      WHERE departureAirportId = 'DEL'
        AND arrivalAirportId = 'BOM'
        AND price >= 1000 AND price <= 5000
      ORDER BY price ASC
      -- with JOINs for Airplane and both Airports

KEY IMPLEMENTATION NOTE — getFlights:
  Unlike other getAll controllers that simply call service.getAll(), the flight
  controller builds a FILTER OBJECT from req.query:

    const filter = {
      departureAirportId: req.query.departureAirportId,    // string
      arrivalAirportId:   req.query.arrivalAirportId,      // string
      minPrice: req.query.minPrice ? parseInt(req.query.minPrice) : undefined,
      maxPrice: req.query.maxPrice ? parseInt(req.query.maxPrice) : undefined,
      departureTime:      req.query.departureTime,         // ISO date string
      sort:               req.query.sort,                  // field name
      order:              req.query.order                  // ASC/DESC
    };

  WHY parseInt? Query params arrive as STRINGS. "5000" !== 5000. Sequelize's
  Op.gte/Op.lte operators need numbers for correct comparison. Without parseInt,
  "5000" > "4500" would be a string comparison (alphabetical), not numeric.

  This is a SUB-RESOURCE endpoint: PATCH /flights/:id/seats
  It only accepts { remainingSeats: 179 } in the body.
  This follows the Principle of Least Privilege — the Booking Service can
  update ONLY remainingSeats, not the flight's price/route/times.

WHAT IS AN API GATEWAY?
  In a microservices architecture, an API Gateway is a single entry point that
  sits in front of all services. It handles cross-cutting concerns like:
    - Authentication & authorization (API keys, JWT tokens)
    - Rate limiting (throttle abusive clients)
    - CORS (cross-origin browser requests)
    - Request tracing (unique IDs for debugging)
    - API documentation (Swagger/OpenAPI)
    - Load balancing & routing to downstream services

  Since we only have ONE service, we baked these features INTO the Flight
  Service rather than creating a separate gateway project.

9.1 CORS — Cross-Origin Resource Sharing
================================================================================

FILE: src/index.js
WHAT: Added `app.use(cors())` to the middleware chain.

WHY:
  Browsers enforce the Same-Origin Policy — a frontend at http://localhost:5173
  cannot fetch from http://localhost:3000/api unless the server explicitly
  allows it with CORS headers. Without CORS, frontend apps get:
    "Access to fetch has been blocked by CORS policy"

HOW IT WORKS:
  The cors() middleware adds these headers to every response:
    Access-Control-Allow-Origin: *          (any origin allowed)
    Access-Control-Allow-Methods: GET,POST,PUT,DELETE,PATCH,OPTIONS
    Access-Control-Allow-Headers: Content-Type, Authorization, x-api-key

  For preflight OPTIONS requests, it responds with 204 No Content + headers.

WHERE IN THE CHAIN:
  Request ID → Helmet → CORS → Rate Limiter → Body Parsers → ...

  CORS goes early so OPTIONS preflight requests are handled before any
  business logic runs.

9.3 API KEY AUTHENTICATION MIDDLEWARE
================================================================================

FILE: src/middlewares/api-key-middleware.js
PURPOSE: Optional authentication gate for all /api/* routes.

LINE-BY-LINE:

  const { serverConfig, Logger } = require('../config');
    → serverConfig.API_KEY reads from process.env.API_KEY (set in .env).

  function validateApiKey(req, res, next) {

    if (!serverConfig.API_KEY) {
        return next();
    }
      → CRITICAL: If API_KEY is empty or unset in .env, the middleware
        does NOTHING — it calls next() and moves on. This means API key
        enforcement is DISABLED by default. You opt in by setting a key.

    const clientKey = req.headers['x-api-key'];
      → Reads the API key from the x-api-key HTTP header.
        Not from query params (insecure — shows in URLs/logs).
        Not from the body (awkward for GET requests).

    if (!clientKey) {
        return res.status(401).json({ ... 'API key is required' ... });
    }
      → 401 Unauthorized: Client sent no key at all.

    if (clientKey !== serverConfig.API_KEY) {
        return res.status(403).json({ ... 'Invalid API key' ... });
    }
      → 403 Forbidden: Client sent a key, but it's wrong.
        Uses constant-time comparison via === (acceptable for
        single-key setups; for multi-key, use crypto.timingSafeEqual).

    next();
      → Key matches — let the request through.
  }

HOW TO USE:
  1. In .env, set: API_KEY=my-secret-key-here
  2. Restart the server
  3. All /api/* requests now require: x-api-key: my-secret-key-here
  4. The dashboard UI at "/" and Swagger at "/api-docs" are NOT gated

WHERE IN THE CHAIN:
  ... → Swagger UI → Static Files → API Key Gate → API Routes → ...

  API key check happens AFTER Swagger and static files are served so
  documentation and the dashboard remain accessible without a key.

9.4 REQUEST ID TRACING
================================================================================

FILE: src/middlewares/request-id-middleware.js
PURPOSE: Assigns a unique ID to every request for log correlation.

LINE-BY-LINE:

  const { v4: uuidv4 } = require('uuid');
    → uuid v4 generates random 128-bit UUIDs like:
      "47e9f16e-6a21-4118-bc64-83c10f481324"

  function attachRequestId(req, res, next) {
    req.id = req.headers['x-request-id'] || uuidv4();
      → If the client already sent an x-request-id header (common in
        microservices where the gateway generates the ID), we REUSE it.
        Otherwise, we generate a new UUID.

    res.setHeader('x-request-id', req.id);
      → Echo the request ID back in the response headers so the client
        can reference it in bug reports or support tickets.

    next();
  }

WHY REQUEST IDs MATTER:
  When you have thousands of requests in your logs, finding the ones
  related to a specific user complaint is impossible without a unique ID.
  
  With request IDs:
    Client sends: POST /api/v1/flights → gets back x-request-id: abc-123
    Client reports: "Flight creation failed, request ID: abc-123"
    Developer searches logs: grep "abc-123" → finds the exact request path

WHERE IN THE CHAIN:
  Request ID → Helmet → CORS → ... (FIRST middleware so every log entry
  downstream can include req.id for correlation)

PHASE 3: COMPLETE REQUEST FLOW EXAMPLES
────────────────────────────────────────────────────────────────────────────────

EXAMPLE 1: Create a City, then create an Airport in that City

  Step 1: POST /api/v1/cities
          Body: { "name": "Mumbai" }
          
          Flow:
          → Middleware checks: name exists? YES ("Mumbai") → next()
          → Controller extracts: { name: "Mumbai" }
          → Service calls: cityRepository.create({ name: "Mumbai" })
          → Repository calls: City.create({ name: "Mumbai" })
          → SQL: INSERT INTO Cities (name, createdAt, updatedAt) VALUES ('Mumbai', NOW(), NOW())
          → Response: 201 Created { id: 1, name: "Mumbai", ... }
  
  Step 2: POST /api/v1/airports
          Body: { "name": "CSMI Airport", "code": "BOM", "address": "Airport Road", "cityId": 1 }
          
          Flow:
          → Middleware checks: name? YES. code? YES. cityId = 1 (positive int)? YES → next()
          → Controller extracts: { name, code, address, cityId }
          → Service calls: airportRepository.create(data)
          → Repository calls: Airport.create(data)  [inherited from CrudRepository]
          → SQL: INSERT INTO Airports (name, code, address, cityId, ...) VALUES ('CSMI Airport', 'BOM', 'Airport Road', 1, ...)
          → MySQL checks: Does cityId=1 exist in Cities? YES → INSERT succeeds
          → Response: 201 Created { id: 1, name: "CSMI Airport", code: "BOM", cityId: 1, ... }

EXAMPLE 2: Fetch an Airport (with City data)

  GET /api/v1/airports/1
  
  Flow:
  → Controller calls: AirportService.getAirport(1)
  → Service calls: airportRepository.get(1)
  → AirportRepository's OVERRIDDEN get() runs (NOT CrudRepository's)
  → Calls: Airport.findByPk(1, { include: [{ model: City, attributes: ['id', 'name'] }] })
  → SQL: SELECT Airports.*, Cities.id, Cities.name FROM Airports
          LEFT OUTER JOIN Cities ON Airports.cityId = Cities.id
          WHERE Airports.id = 1
  → Returns: { id: 1, name: "CSMI Airport", code: "BOM", cityId: 1, City: { id: 1, name: "Mumbai" } }
  → Response: 200 OK with nested City data

EXAMPLE 3: Delete a City (CASCADE deletes airports)

  DELETE /api/v1/cities/1
  
  Flow:
  → Controller calls: CityService.destroyCity(1)
  → Service calls: cityRepository.destroy(1)
  → CrudRepository.destroy(1) calls: City.destroy({ where: { id: 1 } })
  → SQL: DELETE FROM Cities WHERE id = 1
  → MySQL CASCADE: DELETE FROM Airports WHERE cityId = 1 (automatic!)
  → Both the city AND all its airports are gone
  → Response: 200 OK { data: 1 }

EXAMPLE 4: Try to create Airport with non-existent city

  POST /api/v1/airports
  Body: { "name": "Ghost Airport", "code": "GHO", "cityId": 999 }
  
  Flow:
  → Middleware checks: name? YES. code? YES. cityId = 999 (positive int)? YES → next()
     (Middleware can't check if city 999 EXISTS — that requires a DB query)
  → Controller → Service → Repository → Airport.create(data)
  → SQL: INSERT INTO Airports (name, code, cityId, ...) VALUES ('Ghost Airport', 'GHO', 999, ...)
  → MySQL says: FOREIGN KEY CONSTRAINT VIOLATION!
     "Cannot add or update a child row: a foreign key constraint fails"
  → Error is thrown → Repository logs error → Service logs error → Controller catches it
  → Response: 500 Internal Server Error { error: "Cannot add or update..." }

─── ARCHITECTURE: ERROR FLOW (BEFORE vs AFTER) ────────────────────────────────

BEFORE Phase 5:

AFTER Phase 5:

  - Logger.error is called once in globalErrorHandler (not duplicated)


─── SERVICE LAYER: AppError CLASSIFICATION ─────────────────────────────────────

Every service now classifies errors by type:

  ERROR TYPE                       → STATUS CODE
  SequelizeValidationError         → 400 Bad Request
  SequelizeUniqueConstraintError   → 400 Bad Request
  SequelizeForeignKeyConstraintError → 400 Bad Request
  Result is null (not found)       → 404 Not Found
  Update affected 0 rows           → 404 Not Found
  Destroy affected 0 rows          → 404 Not Found
  Everything else                  → 500 Internal Server Error

  PATTERN: Every service catch block does:
    1. Check if error is already AppError → re-throw unchanged
    2. Check if SequelizeValidation/Unique → wrap in AppError(message, 400)
    3. Check if SequelizeFK → wrap in AppError('Referenced X does not exist', 400)
    4. Default → wrap in AppError('Cannot do X', 500)

  WHY 'if (error instanceof AppError) throw error':
    When a not-found AppError is thrown inside the TRY block, it immediately
    enters the CATCH block. Without this guard, the catch would wrap our 404
    inside a new 500 AppError. The instanceof check passes it through unchanged.


─── CONTROLLER LAYER: SIMPLIFICATION ──────────────────────────────────────────

BEFORE (every handler ~40 lines):
  async function getAirplane(req, res) {
    try {
      const airplane = await service.getAirplane(id);
      if (!airplane) {
        Logger.warn('not found', { id });
        return res.status(404).json({...});       ← duplicate null check
      }
      return res.status(200).json({...});
    } catch (error) {
      Logger.error('failed', { error });
      return res.status(500).json({...});          ← always 500
    }
  }

AFTER (every handler ~15 lines):
  async function getAirplane(req, res, next) {
    try {
      const airplane = await service.getAirplane(id);
      return res.status(200).json({...});          ← happy path only
    } catch (error) {
      next(error);                                 ← delegate to global handler
    }
  }

  CHANGES:
  1. Added 'next' as 3rd parameter → (req, res, next)
  2. Removed null/zero checks → service now throws AppError(404)
  3. Removed Logger.error in catch → global handler logs it
  4. Removed res.status(500) catch blocks → next(error) delegates


─── GLOBAL ERROR HANDLER: src/middlewares/global-error-handler.js ─────────────

  function globalErrorHandler(err, req, res, next) {
    Logger.error('Global error handler caught an error', {
      name: err.name, message: err.message, statusCode: err.statusCode,
      path: req.path, method: req.method
    });
    const statusCode = err.statusCode || 500;
    const message = err.explanation || err.message || 'Something went wrong';
    return res.status(statusCode).json({
      success: false, message, data: {},
      error: { statusCode, explanation: message }
    });
  }

  WHY 4 PARAMETERS:
    Express uses the parameter count to identify error handlers.
    3 params = normal middleware: (req, res, next)
    4 params = error handler: (err, req, res, next)
    Even if you don't use 'next', you MUST include it.

  MUST BE REGISTERED AFTER ALL ROUTES:
    app.use('/api', router);           ← routes first
    app.use(globalErrorHandler);        ← error handler last
    If it were before routes, errors would never reach it.

  FALLBACK LOGIC:
    err.statusCode exists → AppError → use it (400, 404, etc.)
    err.statusCode missing → unknown error → default to 500


─── SECURITY: Helmet ──────────────────────────────────────────────────────────

  app.use(helmet());

  Helmet sets ~15 HTTP security headers automatically:
    Content-Security-Policy: default-src 'self'     → prevents XSS via inline scripts
    X-Content-Type-Options: nosniff                 → prevents MIME type sniffing
    X-Frame-Options: SAMEORIGIN                     → prevents clickjacking
    Strict-Transport-Security: max-age=31536000     → forces HTTPS for 1 year
    X-DNS-Prefetch-Control: off                     → prevents DNS prefetch attacks
    X-Download-Options: noopen                      → IE download protection
    Referrer-Policy: no-referrer                    → prevents referrer leaking
    Cross-Origin-Opener-Policy: same-origin         → prevents cross-origin attacks
    Removes X-Powered-By header                     → hides that you use Express


─── SECURITY: Rate Limiting ───────────────────────────────────────────────────

  const limiter = rateLimit({
    windowMs: 15 * 60 * 1000,    // 15-minute window
    max: 100,                     // 100 requests per IP per window
    standardHeaders: true,        // Send RateLimit-* headers
    legacyHeaders: false,
    message: { success: false, message: 'Too many requests...' }
  });
  app.use('/api', limiter);

  HOW IT WORKS:
  - Tracks requests per IP address using an in-memory store
  - After 100 requests in 15 minutes, returns 429 Too Many Requests
  - Sends headers so clients know their remaining quota:
      RateLimit-Limit: 100
      RateLimit-Remaining: 95
      RateLimit-Reset: 767 (seconds until window resets)

  WHY app.use('/api', limiter):
  - Only rate-limits API routes, not static files (if we had any)

16. WHAT COULD GO WRONG — Failure modes and edge cases
================================================================================

1) Partial failure in booking flow
   - Booking created but seat update fails
   - We handle this by rolling back the booking

2) Flight Service down
   - Booking Service returns 503 with meaningful message

3) JWT mismatch between Auth + Gateway
   - Tokens fail to verify if JWT_SECRET differs
   - Fix by keeping secrets aligned in .env

4) Duplicate bookings (race conditions)
   - Two requests could check seats simultaneously
   - Needs DB locking or atomic seat updates in Flight Service

5) Booking cancellation after seat update fails
   - Must ensure flight seat release happens before status change

17. JUNIOR DEVELOPER MISTAKES — Common pitfalls and how to avoid them
================================================================================

1) Hardcoding service URLs
   - Always use env variables (FLIGHT_SERVICE_URL, etc.)

2) Skipping rollback on failures
   - Without rollback, bookings can exist without reserved seats

3) Sharing databases across services
   - Breaks service isolation and makes scaling harder

4) No auth on booking endpoints
   - Allows anyone to create bookings for any user

5) Misordered middleware
   - CORS and rate limiting should run before routes
